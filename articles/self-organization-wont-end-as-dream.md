# FASTに覚悟は必要か？

いきなりですが、これに答えます。

結論、間違いなく必要だと思っています。ただしそれは、現場で必死に頑張るメンバーだけではありません。マネージャー、なんならプロダクト組織の外にも必要なのかもしれないなと最近は思っています。
そして、関わる人達が覚悟を決めるだけでなく、相互に信頼関係を築き、忍耐強く取り組めなければ、確実に破綻すると感じています。

組織として「やりきれるのか？」と問いかけられているような、そんな感覚は僕もありました。

# なぜそう感じたか

FASTでは、メンバーが自分で取り組むタスクを選び、自律的に意思決定します。聞こえは良いですが、実際には非常に重い責任を伴います。

各メンバーに「小さな領域のプロダクトオーナー」のような振る舞いを求められます。自分で選んだタスクに対して、プロダクトマネジメントの観点で判断する必要があるのです。

- なぜこれを今やるべきなのか
- どういった価値が出せるのか
- どれくらいインパクトが見込めるのか

これらを考え抜いた上で、自分の取り組むタスクを選択します。

**しかし、この判断を全員が精度高くできるようになるには、相当な時間と修行が必要です。**

実際にやってみての難しさであったり、そこにどう立ち向かっていったのか？という話については、過去のスクラムフェスなどで登壇している資料があるので、詳しく知りたい方はそちらも見ていただければと思います。

https://speakerdeck.com/wooootack/fast-taught-me-large-scale-agile-hardships-and-fun
https://speakerdeck.com/wooootack/surviving-the-ai-era-lessons-in-a-decentralized-autonomous-organization-with-fast

この高い壁を本当に乗り越える必要があるのか、乗り越えることができると信じて立ち向かえるのか、そんな部分が特に覚悟を求められるなといった印象です。

# 温度差と覚悟の違い

また、柳川さんが記載していた「覚悟と温度差」も、明確に存在していたなと思います。

というのも、FASTを導入して1年くらいが経過した頃に、社内で50名ほどを対象にアンケートを取ったことがあります。 詳細なアンケート結果については、こちらのスライドに記載しているため、興味があれば目を通していただければと思います。
https://speakerdeck.com/wooootack/review-of-the-first-year-of-fast-implementation

職種ごとの差や、FAST歴による違いなどは当然あったのですが、FAST歴が長くても理解度や納得度が高くなかったりと、やはり認識が揃いきっていない部分は感じ取れました。 そして、FASTに対するスタンスや理解度は、もはや一概に職種や経験、年齢などでカテゴライズできるものではないなと感じました。

カテゴライズをせずに個人と向き合い「この覚悟の違いと温度差」を埋めていくことこそが「自己組織化」を推進すると決めたからにはやりきらねばならないことだと感じています。

# 自己組織化との向き合い方

とはいえ、別にエンジニアリングマネージャーでもスクラムマスターでもアジャイルコーチでもなんでもない僕が、なぜこういった思想なのかもせっかくなので言語化してみようと思います。

実は、僕はFASTの導入には当初あまり乗り気ではありませんでした。というのも、僕の所属していたスクラムチームはとても良いチームだったからです。なぜこんなに良いチームを壊すんだろう？と本気で思っていました。

しかし、実はチームの外では既に多くの問題が発生していました。そして、まだそこに僕は向き合えていませんでした。 このような経緯もあり、僕は少しずつチームの外にも目を向けるようになっていきました。

（このあたりの導入背景などはよければこちらもどうぞ）

https://speakerdeck.com/yoshikiiida/rsgt2025

チームの外に目を向けると、今まで問題と思っていなかったところが問題なんだと気がつけるようになり、自分の影響を与えるべき範囲が、チームから組織に広がりました。 そして、FASTであれば、その問題に対して自らダイレクトに解決策を取ることが可能です。なぜなら、我々はコレクティブという大きな塊の一員であり、我々のやるべきことは、コレクティブの目標達成だからです。

一方で、今までよりも難易度は上がりました。人数も増えて規模も大きくなったし、多種多様な人がいるし、これまで長くチームで一緒に活動をしたわけではないので、お互いの解像度も低い状態です。

さらに、その頃から少しずつアジャイルやスクラムのコミュニティに積極的に参加するようになったことも一因だと思っています。そこでは、僕より経験も知識も豊富な人がたくさんいて、常に多くの学びと刺激を与えてくれます。

そして最後が、僕が本当の意味でやりたいことが、FASTという手段を通じて組織づくりに貢献し、その組織で良いプロダクトを作り出した先にあると思ったからです。そのためにも、自己組織化の可能性を信じてみようと思いました。

振り返ると、4つの要素がうまくハマったからこそ「自己組織化」に真正面から向き合えるようになったのだと思います。

- 活動範囲の広がり：チームから組織へ
- これまでにない難易度の挑戦：規模も多様性も増した環境
- 挑戦を支える学びと刺激：コミュニティでの出会い
- 自分のやりたいこと：良い組織で良いプロダクトを作り出すという目標

# 自己組織化と真面目に向き合った先にあったもの

そんなこんなで「自己組織化」に向けて、新たな挑戦をするわけですが、楽しくもあり、苦労も多かったです。

同じ会社に所属して、同じミッションに共感し、仕事をしていても、どうしても目先の目標や持っているコンテキストの違いなどからコンフリクトが起きたり、自分の考えや想いを伝えきれずに、悔しいなと感じるときもありました。

常に全員が同じ方向に進むことも、難しくなりましたし、それを受け入れる必要があることも理解しました。ただ安定させるのではなく、多少のカオスが組織を強くすることも学びました。

影響範囲が広まったこともあり、ステークホルダーマネジメントの重要性も認識できました。自分たちだけが勝てる「ゼロサムゲーム」ではなく、全員が勝てる「プラスサムゲーム」へと考え方の変化が求められました。

総じて、チームでやっていた頃は、僕はあまり経験したことがないことだらけでした。当然、知識も経験もない領域の問題解決は、楽しくもあり難しくもありました。その過程でさまざまなより広範囲な組織に関する知識も少しずつ身につきました。（一部は過去のZennで記事に書いたりもしているので、興味があれば漁ってみてください）

その一方で、こういった知識を手に入れたとしても、まだ僕にそれを実現するだけの力がないなということも同時に実感しました。僕の成長を実感していながらも、やはり経験や実績が足りないと感じることもあります。人の心を突き動かすようなリーダーシップも不足していると感じることがあります。

これは「夢物語」と言われてしまうような「自己組織化」という高い目標に向けて挑戦するという覚悟を決めたからこその成長と悔しさなんだろうと感じています。

# 結局「自己組織化」できたのか

正直、まだまだ道半ばだなと思っています。それと同時に「やり切れました」って言える日は来ないだろうなとも思っています。

当たり前ですが「自己組織化」をすることがゴールではなく、あくまで自分の作りたい価値のために、自己組織化という手段の相性が良いから、これを実現したいと考えているに過ぎません。自己組織化というものを前向きに断念する未来もあるかもしれません。

また、ひとえに「自己組織化」と言っても、会社のフェーズ、メンバーの成熟度、ビジネスの状況などで、目指すべき状態としては大きく異なると思っています。そういった状況を加味した上で、組織として今はこういう状態を目指すべきで、将来的にはこういう状態を目指すべきだというストーリーを描き、みんなを導いていけるような存在になれると良いなと思っています。

# さいごに

冒頭でも書いたように、僕は「夢物語だと思っているうちは、夢物語にしかならない」と思っています。 何事もそうですが、夢を現実にできるのは、実現できると信じた人だけだと信じています。（こんなカッコつけたことを言っていますが、僕も何度も挫けそうになっています）

そして何より、自己組織化は夢物語などではなく、今後の時代もプロダクトづくりで生き残っていきたいのであれば、必須のスキルとなっていくと思っています。僕がなぜそう思うのか？は、以下の記事の内容も参考になるかと思います。

https://zenn.dev/loglass/articles/agile-as-management-issue-hcm-fast

「自己組織化」は「夢物語」と言われるほど、難易度が高く難しいものだと思います。そんな高い壁に向かっていく途中、どこからともなく「それ夢物語だから、そろそろ現実を見なよ」という声も聞こえてくるかもしれません。でも、そんな甘い囁きに負けるわけにはいかないのです。覚悟を持って、これからも試行錯誤を積み重ねていきたいと思っています。

**「夢物語だと思っているうちは、夢物語にしかならない！俺が現実にするんだ！」**
